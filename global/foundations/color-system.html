Color System

MUI's palette token system is the foundation for all Colorkrew products. Color decisions reference semantic tokens — never raw hex values. Each product defines its full palette in _theme.md.

Token Format

All color tokens follow MUI's palette naming convention:

palette.<role>.<variant> — e.g., palette.primary.main, palette.text.secondary

Never reference a hex value directly in a component spec or design token file. Always use the semantic name so that themes (light/dark) and product overrides work correctly.

Required Palette Roles

Every product must define all of the following roles in its _theme.md.

Role Variants Typical Use
primary main, light, dark, contrastText Primary actions, links, key highlights
secondary main, light, dark, contrastText Secondary actions, accents
error main, light, dark, contrastText Errors, destructive actions, validation failures
warning main, light, dark, contrastText Warnings, cautionary states
info main, light, dark, contrastText Informational messages, tips
success main, light, dark, contrastText Confirmation, completed states, positive feedback
text primary, secondary, disabled All text content
background default, paper Page background, surface backgrounds (cards, modals)
action active, hover, selected, disabled, disabledBackground, focus Interactive element states
divider (single value) Separators, borders between sections

Brand & Custom Tokens

Products may add brand-specific color tokens using the same format. Define them under a descriptive role name:

palette.brand.main, palette.brand.light, palette.brand.dark, palette.brand.contrastText

All custom tokens must be listed and documented in the product's _theme.md.

What Goes in Each Product's _theme.md

  • All required palette roles above with resolved hex values
  • Any additional brand-specific tokens
  • Dark mode palette overrides, clearly labeled and separated from light mode values
  • Notes on intended usage context per token (e.g., "primary.light is used for hover backgrounds only")
Accessibility reminder: All text/background color combinations must meet WCAG 2.1 AA contrast minimums (4.5:1 for normal text, 3:1 for large text and UI components). Verify contrast before finalizing palette values. See Accessibility for full requirements.

Do / Don't

Do
  • Reference tokens by name: palette.primary.main
  • Define all required roles in _theme.md
  • Document any custom tokens with usage context
  • Label dark mode values separately
  • Verify contrast ratios against WCAG 2.1 AA
Don't
  • Hardcode hex or RGB values in component specs
  • Invent token names outside MUI convention without documenting them
  • Apply primary.main for decorative non-interactive elements
  • Mix light and dark mode values without clear labeling
